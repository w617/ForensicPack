# Release Artifact Policy

## Goal

Keep source history readable while still producing distributable Windows artifacts.

## Policy

- Source of truth is the Python code in `src/`.
- Official distributables are GitHub Release assets generated by CI.
- `release/windows/` is treated as vendored/runtime content for repository analytics.

## Recommended Workflow

1. Develop and test in `src/`.
2. Merge changes to `main` after CI passes.
3. Create a version tag (`vX.Y.Z`).
4. Let `release-windows.yml` build and publish ZIP artifacts.
5. Publish checksums in `checksums/` as needed.

## Notes

- Keeping large runtime bundles in git can increase clone time.
- If you later choose source-only git history, remove tracked `release/windows/` artifacts and rely only on Releases.
